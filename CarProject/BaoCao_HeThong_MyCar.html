Tổng Quan Hệ Thống

MyCar Showroom là hệ thống quản lý showroom xe hơi cao cấp, được xây dựng trên nền tảng ASP.NET Core 10.0 sử dụng Razor Pages. Hệ thống cung cấp giao diện người dùng sang trọng phong cách Mercedes-Benz kết hợp với quản trị nội dung CRUD đầy đủ cho Admin, dashboard doanh thu cho Quản Lý showroom, và trải nghiệm mua sắm xe hơi trực tuyến cho người dùng cuối.

Xác Thực và Phân Quyền

3.1. Cơ chế đăng nhập

Hệ thống hỗ trợ hai cơ chế xác thực song song:

1. Session Cookie (Razor Pages)
Sau khi đăng nhập thành công, thông tin người dùng (UserName, UserRole, AvatarUrl, Email, MaTaiKhoan) được lưu vào session server-side. Session cookie HttpOnly có thời gian sống 30 phút. Khi logout, session được xóa và cookie bị hủy.

2. JWT Bearer (API Clients)
Endpoint POST /api/login nhận { tenDangNhap, matKhau } và trả về JWT token (HMAC-SHA256, 30 phút). Token chứa claims: NameIdentifier, Name, Role, AvatarUrl, Email, MaTaiKhoan. Dùng cho React frontend hoặc client API.

3. Google OAuth 2.0
Đăng nhập qua Google (callback /Account/GoogleCallback). Nếu email chưa tồn tại trong DB, tự động tạo tài khoản mới với vai trò User. Sau đăng nhập, chuyển hướng theo vai trò (Admin → /Admin/Index, Quản Lý → /QuanLy/Dashboard, User → /Index).

3.2. Phân quyền

Vai tròQuyền hạn
AdminTruy cập toàn bộ /Admin/* và tất cả CRUD (13 chức năng)
Quản LýTruy cập /QuanLy/Dashboard — xem doanh thu showroom được phân công

Cơ chế bảo vệ path-based authorization:

  • /Admin/* — chỉ Admin mới truy cập được (kiểm tra Session Role == "Admin")
  • /Orders/*, /Profile, /TestDrive/*, /QuanLy/* — yêu cầu đăng nhập
  • Nếu không có quyền: trả về 403 (Admin) hoặc redirect đến /Account/Login

Kiến Trúc Docker & Luồng Bảo Mật

9.1. Docker — SQL Server hoạt động thế nào?

Trước khi có Docker: Phải cài SQL Server trực tiếp trên Windows — nặng ~2GB, ngốn RAM, xung đột với các SQL Server khác, gỡ cài đặt phức tạp.

Docker giải quyết bằng container: Docker chạy SQL Server trong một container — giống máy ảo siêu nhẹ, chỉ chạy đúng một ứng dụng (SQL Server), cô lập hoàn toàn với hệ thống.

┌──────────────────────────────────────────────────┐
│                Máy Windows                         │
│                                                    │
│  ┌─────────────────────┐   ┌──────────────────┐   │
│  │ Container:           │   │ CarProject.exe   │   │
│  │ sqlserver-carproject │   │ (Web App)        │   │
│  │                      │   │                  │   │
│  │ SQL Server 2022      │   │ ASP.NET Core     │   │
│  │ Port 1433 (nội bộ)   │   │ Port 5001        │   │
│  └─────────┬────────────┘   └──────────────────┘   │
│            │                                       │
│            ▼                                       │
│     localhost:1433                                  │
└────────────────────────────────────────────────────┘

Các bước chi tiết:

BướcHành độngGiải thích
1Pull image SQL Serverdocker pull mcr.microsoft.com/mssql/server:2022-latest — tải image SQL Server 2022 từ Microsoft (1.5GB)
2Tạo containerdocker run -e "ACCEPT_EULA=Y" -e "SA_PASSWORD=..." -p 1433:1433 --name sqlserver-carproject -d mcr.microsoft.com/mssql/server:2022-latest
3Port mappingCổng 1433 của máy chủ → cổng 1433 trong container. Kết nối qua localhost,1433 hoặc 127.0.0.1,1433
4Web app kết nốiEF Core mở kết nối TCP tới 100.108.48.1,1433 (IP Tailscale cho phép truy cập từ xa), gửi câu SQL parameterized
Tại sao dùng 100.108.48.1 thay vì localhost? Đó là IP Tailscale — cho phép kết nối từ laptop khác trong cùng mạng Tailscale. Khi chạy local hoàn toàn, có thể dùng localhost,1433.

So sánh Docker vs Cài đặt thường:

Khía cạnhCài SQL Server trực tiếpDocker container
Dung lượng2-3 GB1.5 GB (image)
RAMNgốn RAM mặc địnhGiới hạn được (--memory)
Xung độtDễ xung đột portCô lập hoàn toàn
Gỡ cài đặtPhức tạp, để lại registrydocker rm là sạch
Nhiều phiên bảnKhó, cần nhiều portDễ — mỗi version 1 container

9.2. Luồng xác thực (Authentication)

Nguoi dung gui: tenDangNhap + matKhau
         │
         ▼
  Login.cshtml.OnPostAsync()

         │

         │   - Session.SetString("UserName", ...)
         │   - Session.SetString("UserRole", ...)    // Admin | Quan Ly | User

         │

            Admin    → /Admin/Index
            Quan Ly  → /QuanLy/Dashboard
            User     → /Index

9.3. Cơ chế Session

Khi dang nhap thanh cong:
         │
         ├─ ASP.NET Core tao Session ID (GUID ngau nhien)
         ├─ Gui cookie HttpOnly ve trinh duyet:
         │   Set-Cookie: .CarProject.Session=abc123...; path=/; httponly
         │
         ├─ Du lieu session luu tren SERVER (bo nho RAM)
         │   Khong gui du lieu nhay cam xuong client
         │
         ├─ Moi request sau do:
         │   Browser gui cookie → Server tra Session ID
         │   → Doc du lieu tu bo nho server
         │
         └─ Timeout: 30 phut khong hoat dong → tu dong xoa session

HttpOnly là gì? Cookie có flag HttpOnly = true — JavaScript không thể đọc cookie này. Chống tấn công XSS (khi hacker chèn script vào trang, script không lấy được session cookie).

9.4. Phân quyền (Authorization)

Middleware tùy chỉnh kiểm tra từng request theo path:

Moi request den
         │
         ▼
  Custom middleware kiem tra path:
         │
         ├─ Neu path bat dau bang /Admin/:
         │   ├─ Session co UserName? → Neu khong → redirect /Account/Login
         │   └─ UserRole == "Admin"? → Neu khong → tra ve 403 Forbidden
         │
         ├─ Neu path bat dau bang /QuanLy/, /Orders/, /Profile/, /TestDrive/:
         │   └─ Session co UserName? → Neu khong → redirect /Account/Login
         │
         └─ Cac path khac: cho phep truy cap (public)

9.5. JWT cho API

POST /api/login { tenDangNhap, matKhau }
         │
         ├─ Xac thuc giong login form
         ├─ Tao JWT token:
         │   Header:  { alg: "HS256" }
         │   Payload: { sub: "MaTaiKhoan", name: "TenDangNhap",
         │              role: "UserRole", email: "Email" }
         │   Sign:    HMAC-SHA256(Header.Payload, SecretKey)
         │            SecretKey = "CarProjectSuperSecretKey2024@..."
         │
         └─ Tra ve: { token: "eyJ...", expiresIn: 1800 }

Token có hạn 30 phút. Client gửi token trong header Authorization: Bearer eyJ.... Server verify chữ ký HMAC để đảm bảo token không bị giả mạo.

9.6. Google OAuth 2.0

Nguoi dung → Click "Dang nhap Google"
         │
         ├─ Redirect den: accounts.google.com/o/oauth2/auth
         │   Kem: ClientId, RedirectUri, Scope(email, profile)
         │
         ├─ Google hien thi man hinh dang nhap + xac nhan
         │
         ├─ Nguoi dung xac nhan → Google redirect ve:
         │   /Account/GoogleCallback?code=...
         │
         ├─ Server trao doi code lay access_token
         ├─ Server dung token goi Google API lay email + ten
         │
         ├─ Neu email chua co trong DB:
         │   Tao tai khoan moi (VaiTro = "User", TrangThai = "Active")
         │
         └─ Ghi session + redirect nhu login thuong
